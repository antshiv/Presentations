The rule is simple: if any input variable changes between your baseline run and your optimized run, the performance delta is noise. Use the one-time command live; use persistent config on dedicated profiling machines.

Before any profiler can collect meaningful data, the Linux kernel has to let it. By default, Linux is conservative about who can access hardware performance counters, kernel symbols, and process address spaces. This slide shows you how to open those gates. Three sysctl values matter. First: kernel.perf_event_paranoid. Setting this to 1 allows non-root processes to access the hardware performance monitoring units — the PMUs that count cycles, instructions, cache misses, and branch misses. Setting it to -1 allows everything including kernel sampling, but that is a security concern on shared machines. Value 1 is the right default for a developer workstation. Second: kernel.kptr_restrict. This controls whether /proc/kallsyms — the kernel symbol table — is readable by non-root users. Setting it to 0 lets perf and VTune resolve kernel function names in stack traces. Without this, you will see bracketed-unknown entries in your FlameGraph wherever kernel code ran: interrupt handlers, scheduler functions, system calls. These obscure the real picture. Third: kernel.yama.ptrace_scope. This controls whether one process can attach to and trace another. VTune's collection daemon needs to attach to your inference process. Setting ptrace_scope to 0 enables this. With value 1 (the default on most distros), VTune collection silently fails or runs in a degraded mode. The persistent block at the bottom writes these to /etc/sysctl.d/99-ck-profiling.conf. On a dedicated profiling workstation, make this permanent so you do not need to remember the sysctl commands each session. On a managed corporate machine, you may not be able to change these values — in that case, use the pre-captured artifact fallback path shown later. Important: these permission changes do not affect running processes, only new profiling sessions. No restart needed.

This is where everything starts. Step 8 is the canonical entry point to the v7 inference pipeline. The ck_run_v7.py script takes the HuggingFace GGUF URI as its first argument. The hf:// prefix means it checks the model cache first, downloads if the file is missing, or uses the local cached copy. Qwen3-0.6B-Q8_0.gguf is the full 8-bit integer quantized version. Q8_0 means each weight value is stored as an 8-bit integer with a per-block floating-point scale factor. At 0.6 billion parameters at 8-bit, the file is roughly 620 megabytes. The --context-len 1024 sets the KV cache allocation. This matters for profiling because it determines how much memory is pre-allocated for the attention cache. Larger context means more allocation and potentially different cache behavior during decode. The --force-compile flag rebuilds libmodel.so from the generated C even if the binary already exists. Always use this before profiling runs — you want fresh symbols, no stale binary from a previous optimization experiment that might have had different code. The --force-convert triggers the full IR pipeline: it reads the GGUF, generates ir1_embedding.json and other per-layer IR files, runs the lowering passes for scheduling and memory layout, and emits model_v7.c. Without this flag, cached IR artifacts might not reflect the current codegen logic. The --generate-visualizer flag creates the initial ir_report.html. It will be mostly empty of profiling data at this stage, but the structure will be there with placeholder cards. Let me walk through it left to right in the pipeline. Qwen3-0.6B-Q8_0.gguf is the source. This never changes — it is the downloaded model file. The lowered_ files are after the lowering passes: scheduling (what order to execute operations), memory layout (where each tensor buffer lives, alignment, whether it is contiguous), and type specialization. layout.json captures the final memory layout decisions. model_v7.c is the generated C source. This is a real file you can open and read. It contains the full forward pass logic with AVX intrinsic calls inlined. If a kernel is behaving unexpectedly in the flamegraph, you can open this file and find the exact C code that compiled to that function. libmodel.so is the shared library compiled from model_v7.c. weights.bump is the contiguous binary blob of all model weights, packed in a single allocation. The .ck_build directory contains compiled object files and the build artifacts. The echo at the end prints the pipeline summary: GGUF to IR to generated C to libmodel.so. Codegen Infographic

Codegen path used by Step 8: GGUF -> IR -> generated C -> libmodel.so

This infographic is the visual version of the artifact chain we just traced. Let me walk through each stage so the architecture is clear before we profile anything. Stage 1 — GGUF Source. The input is Qwen3-0.6B-Q8_0.gguf. GGUF is the standard format for distributing quantized models on HuggingFace. It packs model weights in a custom quantization format along with the architecture metadata — layer counts, hidden dimensions, attention head configuration, all of it. Stage 2 — IR Pipeline. The GGUF reader extracts each layer's weights and architecture info, and generates the ir1_* JSON files. These are the first-pass intermediate representations. Then the lowering pipeline applies three transformation passes: type inference (resolving what data types flow between operations), op scheduling (determining computation order to minimize memory footprint), and memory layout analysis (deciding buffer placement, alignment, and whether weight tiles can be kept contiguous for cache-friendly access). The output is layout.json and the lowered_* files. Stage 3 — Code Generation. The code generator reads the lowered IR and emits model_v7.c with AVX intrinsic calls for the hot paths. Critically, the codegen detects the CPU's ISA features at generation time using CPUID and selects the best kernel variant. On this Alderlake-P, it selects AVX2 variants — not AVX-512, which is absent. Stage 4 — Compile. GCC or Clang compiles model_v7.c at -O3 -march=native. The -march=native flag tells the compiler to use every SIMD feature your specific CPU supports. On this machine, that ceiling is AVX2. Stage 5-6 — Runtime. libmodel.so is loaded via dlopen() at inference time. ck-cli-v7 invokes it for decode and prefill inference. This is the binary we profile. The key insight: every optimization has a stage where it lives. Want better SIMD? That is stages 3 and 4. Want better memory layout? That is stage 2. This makes the system surgically debuggable.

This table is the decision tree for which profiling tool to reach for first. Starting with the wrong tool is how you spend an hour collecting data that answers the wrong question. Row 1: You observe low IPC and high cache misses. IPC below 2.0 combined with LLC miss rate above 30% is a strong signal of memory pressure. The right first tool is perf stat because it confirms or refutes that signal in under 30 seconds. You do not need VTune's heavy collection overhead for this first-pass triage. Row 2: Something is slow but you do not know which function. Reach for FlameGraph. It gives you the call stack decomposition in a visual format where the widest frame immediately reveals the hotspot. For our workload, the widest frame turns out to be vec_dot_q8_0_q8_0_avx consuming 69% of decode CPU time — immediately visible as the dominant orange block. Row 3: You know there are high LLC misses but you are not sure whether the bottleneck is bandwidth-limited or latency-limited. VTune's Memory Access analysis is the right tool. Counters to track

  • Cache miss rates (LLC)

Interpretation

  • Low IPC + high misses → memory pressure.
  • High dTLB miss/page-walk rate → address-translation pressure.

  • High stalls → scheduling or data starvation.
Let me spend a few slides on what perf stat actually measures, because misreading these counters is one of the most common mistakes in performance work. IPC — Instructions Per Cycle — is your first gut-check number. A modern out-of-order CPU can theoretically retire 4 to 6 instructions per cycle per core when the pipeline is well-fed. If you see IPC below 1.0, the processor is stalling heavily — almost certainly on memory. Between 1.5 and 2.5 is typical for mixed workloads. Above 3.0 means you are feeding the pipeline well. Our 2.21 is in the middle — not alarming, but there are stalls. Cache miss rates need to be interpreted at each hierarchy level. L1 misses cost about 4 cycles (data comes from L2). L2 misses cost about 12 cycles (data from L3). L3 or LLC misses cost 200 to 300 cycles — the data comes from DRAM, and depending on whether the DRAM row buffer is open, it can take even longer. At 38.6% LLC miss rate, we are sending nearly 4 in 10 LLC requests to DRAM at 200+ cycle cost. That is the dominant source of stall cycles in the decode path. dTLB misses are a separate concern from cache misses. A dTLB miss does not mean your data is not in cache — it means the CPU does not know where in physical memory to look, because the virtual-to-physical address translation is not cached in the Translation Lookaside Buffer. A TLB miss triggers a hardware page-table walk through PGD, PUD, PMD, and PTE levels. In the worst case with 4-level page tables and cold cache, a single TLB miss can cost 4 LLC misses worth of latency. Our walk_completed count of 6.1 million is in the watch zone — not yet critical, but worth monitoring. Branch miss rate tells you about control flow efficiency. For tight SIMD loops like GEMV or GEMM kernels, branch misses should be near zero because the loop structure is simple and predictable. Our 0.25% branch miss rate confirms the hot kernels are branch-light. Backend stalls is the catch-all for all execution unit and memory-wait cycles. TLB Infographic

TLB hit vs miss intuition: page-walk overhead is the cost behind high translation-miss counters.

This infographic gives you the visual intuition for why TLB misses matter — and they matter more than most engineers expect. Follow the hit path first: your CPU executes a load instruction that accesses a virtual address. The hardware MMU (Memory Management Unit) checks the L1 dTLB (data translation lookaside buffer) — a small, fast cache holding 64 virtual-to-physical address mappings. If the mapping is there, you get the physical address in 1 cycle and the actual cache lookup can proceed immediately. This is the happy path. Now the miss path: the L1 dTLB does not have the mapping. The hardware then checks the L2 STLB (Shared TLB) — larger, holding about 1,500 entries, shared between instruction and data TLBs. If it is there, you pay 7 to 10 extra cycles. Manageable. If the STLB also misses, the hardware page-table walker kicks in and traverses the 4-level page table hierarchy: PGD, PUD, PMD, PTE. Each level is a memory access. In the worst case, all four are cache misses. Four LLC misses at 200 cycles each is 800 cycles of translation overhead just to find out where your data lives. Here is the perspective check: a tight AVX2 GEMV inner loop can process 32 int8 dot-product multiply-accumulates per cycle. An 800-cycle page-table walk is equivalent to throwing away 25,600 multiply-accumulate operations. That is the real cost of bad locality. The C-Kernel-Engine uses a contiguous weights.bump allocation precisely to avoid this. All model weights are packed in one contiguous block. This means the page access pattern during inference follows a predictable sequential stride, which keeps the hardware prefetcher effective and TLB miss rates manageable. Our 6.1 million walks over the entire inference run reflects this design.

This infographic provides the hardware memory hierarchy context for why cache misses and access latency differ depending on your machine architecture. On a multi-socket server — which is typical for cloud inference deployment — you have multiple NUMA nodes. Each socket has its own processor cores, its own L3 cache, and its own DRAM. When a core on socket 0 requests data that was allocated by a process running on socket 1, the data must travel over the inter-socket interconnect (called UPI on modern Intel Xeon). How to Read FlameGraph Correctly

Core semantics

  • X-axis: grouped stack space, not wall-clock timeline.
  • Width: cumulative sampled CPU time (inclusive).
  • Y-axis: call depth from caller to callee.
  • Color: usually aesthetic grouping, not cost by itself.

Operator reading rule

  • Start with the widest boxes near the top of heavy stacks.
  • Trace downward to find the parent path that creates the cost.
  • Ignore narrow spikes until wide hotspots are improved.
  • Compare decode vs prefill; they often have different bottlenecks.
Before I show you the actual flamegraph from this machine, let me make sure you can read it correctly. Flamegraphs are frequently misread, and misreading one sends you optimizing the wrong function. The X-axis is NOT time. I want to say this twice because it is the most common misread: the horizontal axis in a flamegraph is NOT wall-clock time flowing left to right. It is aggregated stack space. The width of a box represents how many CPU samples were collected with that function somewhere in the call stack. Functions are sorted alphabetically within the same call depth level, not by when they executed. This means: vec_dot_q8_0_q8_0_avx appearing as a wide orange block does not mean it ran for a long contiguous time. It means that across all the random samples taken during profiling, this function was present in the call stack often enough to account for 69% of all samples. Width equals frequency. The Y-axis IS call depth. The bottom of the graph is typically main or a thread entry point. Each layer up is a function called by the layer below it. The leaf frames — functions that were executing without calling anything else — appear at the very top. Leaf frames are where the CPU is actually doing computational work, as opposed to just waiting in a call to something deeper. Color is typically aesthetic. In Brendan Gregg's standard flamegraph tool, colors are random and used only to visually distinguish adjacent boxes. Some custom flamegraph tools use color to indicate kernel vs userspace vs JIT code, but do not interpret random color variation as meaningful cost variation. The practical reading rule: start with the widest boxes near the top of the heaviest stacks. That is your optimization target. Trace downward to understand the call path that creates the cost. Compare decode vs prefill graphs — the dominant functions are often completely different, which tells you the two modes have different bottleneck types. Ignore narrow spikes and tall thin towers until the wide dominant frames are optimized. The narrow stuff is the noise.

Advisor: Roofline and SIMD Headroom

Why Advisor matters

  • Arithmetic intensity view.
  • Compute roof vs memory roof position.
  • Vectorization opportunity visibility.

Typical actionables

  • Improve data reuse / blocking.
  • Strengthen SIMD utilization.
  • Reduce scalar fallback paths.
Intel Advisor's roofline model answers a question that perf stat and VTune alone cannot fully answer: even if you fix the immediate bottleneck, how much headroom is there for improvement? The roofline model plots two things against each other. X-axis: arithmetic intensity, measured in FLOPs or integer operations per byte of data moved from memory. Low arithmetic intensity means you do few calculations per byte — this is the signature of a memory-bound workload. High arithmetic intensity means you do lots of math per byte — this is compute-bound territory. Y-axis: achieved compute throughput, measured in GFLOPs per second or GOPS per second for integer operations. Higher is better. Two lines create the roofline boundaries. The memory bandwidth roof has a positive slope from lower-left to upper-right: it equals peak_memory_bandwidth × arithmetic_intensity. Any kernel below this line is memory-bound — it cannot go faster without either increasing arithmetic intensity or increasing memory bandwidth. The compute roof is horizontal: it equals peak_FLOP_rate of the processor. Kernels below the compute roof but above the memory roof are compute-bound. For our Alderlake-P machine: at 33 GB/s DRAM bandwidth and 8-bit integer operations, the memory roof slope gives a rough ceiling around 264 GOPS per second for memory-bound workloads. The AVX2 compute peak for int8 operations across 8 P-cores is approximately 512 GOPS per second (8 cores × 3.5 GHz × 16 int8 per YMM × 2 for multiply-accumulate). The ratio between compute and memory ceilings is about 2:1 — which means memory-bound kernels have theoretical room to double throughput if arithmetic intensity can be doubled. For vec_dot in decode: arithmetic intensity is roughly 1 operation per byte — far left of the ridge. Memory-bound. For gemm_nt in prefill with batch reuse: arithmetic intensity is much higher — closer to compute-bound.

Roofline Infographic

Roofline mental model used on Advisor slide: move right (intensity) and up (throughput) toward the compute roof.

Let me give you the complete mental model for reading a roofline chart, because it is one of the most powerful tools in HPC performance analysis and it is frequently misread or underused. Imagine a two-dimensional space. The horizontal axis — arithmetic intensity — describes how efficiently your algorithm reuses data relative to how much data it loads. A simple copy operation has AI of 0 (it loads data and stores it without doing any math). A dense matrix multiply where weights are heavily reused can have AI of 100 or more. The vertical axis — performance — is how fast your kernel actually runs, in floating-point or integer operations per second. Higher is always better. Two performance ceilings bound what is achievable. The memory bandwidth ceiling is a diagonal line from lower-left to upper-right: performance = bandwidth × arithmetic_intensity. Below this line, you are memory-bound. Your computation rate is limited by how fast you can fetch data from DRAM. The only way to improve: either increase arithmetic intensity (reuse data more) or move to faster memory. The compute ceiling is horizontal: performance = peak_FLOP_rate. Below this ceiling but above the memory line, you are compute-bound. Your computation rate is limited by the execution units. The only way to improve: better instruction mix, wider SIMD, reduced overhead. The ridge point is where the two ceilings meet. It is the sweet spot: here, memory bandwidth and compute throughput are perfectly balanced. The goal of cache-blocking optimizations is to move memory-bound kernels rightward (increase AI by exploiting L2/L3 reuse) until they approach or cross the ridge. For vec_dot in decode mode: the weight matrix is read once per token, used once, and discarded. That is very low arithmetic intensity — far left. Memory-bound. For gemm_nt in prefill: each weight tile is reused across all N input tokens. Higher arithmetic intensity — further right, closer to the ridge. This roofline view quantifies exactly why decode is slow and prefill is fast.

Goal

Avoid oversubscription and thread contention.

Validation

Check VTune thread timeline and CPU utilization.

Thread count is a lever that can dominate performance measurement variance if not controlled carefully. On the Alderlake-P chip with 8 P-cores and 8 E-cores, we set OMP_NUM_THREADS=12 for a deliberate reason. Using all 16 logical cores means E-cores participate in the thread pool. E-cores run at lower maximum frequency and have half the L2 cache of P-cores (512 KB vs 1.25 MB per core). At 12 threads, the thread pool primarily fills the 8 P-cores with room for load balancing. At 16 threads, E-cores contribute to the pool and can introduce timing variance — a decode step where a critical kernel thread lands on an E-core is slower than when it lands on a P-core. MKL_NUM_THREADS=1 prevents Intel Math Kernel Library from spawning its own internal thread pool. Some system Python environments link NumPy against Intel MKL, and if the Python runner script for the profiling pipeline inadvertently triggers an MKL operation, it will spawn up to 16 threads to handle it — which can saturate the CPU and compete with the OMP pool. Setting MKL to single-threaded makes the Python components non-interfering. OPENBLAS_NUM_THREADS=1 is the same story for OpenBLAS. These two lines are defensive: they prevent external library thread pools from racing with the controlled OMP pool and causing noise in the profiling results. To validate threading in VTune: open the Threading view and look at the CPU utilization per thread timeline. You want to see approximately 12 active threads during the inference window, all with similar utilization. If you see 2 to 3 busy threads and 9 idle ones, the thread pool is not distributing work correctly — investigate the dispatch logic. If you see 16 threads competing and high context-switch overhead, reduce the thread count.

DRAM Pressure vs Arithmetic Intensity

Memory-bound symptom

High LLC misses + backend stalls + low IPC.

Compute-bound symptom

High IPC, lower miss pressure, ALU/FMA saturated.

Fix class A

Blocking, layout, prefetch, cache-friendly ordering.

Fix class B

Kernel fusion, SIMD widening, unroll, reduced overhead.

This framework unifies everything we have measured into a single conceptual model. The memory-bound symptoms on our machine align perfectly: LLC miss rate 38.6% is high, indicating data is not being reused from cache. DRAM bandwidth averages 17.6 GB/s with peaks at 31.9 GB/s near the 33 GB/s hardware limit. VTune Back-End Bound is 45.4% with Memory Bound at 21.2%. These all point to the same root cause: the working set of active data during decode — primarily the weight matrix being multiplied by the current hidden state — exceeds the L3 cache capacity and streams from DRAM on every token step. The vec_dot function has very low arithmetic intensity. For each 32-byte chunk it loads from memory, it performs 32 multiply-accumulate operations. That is 32 integer MACs per 32 bytes loaded, which is 1 operation per byte — extremely low. At 33 GB/s bandwidth limit and 1 op/byte, the theoretical maximum is 33 GOPS per second. In practice, with LLC miss overhead and DRAM latency, we achieve significantly less. The compute-bound symptoms for prefill: gemm_nt processes each weight byte multiple times across the full token batch. With N=64 input tokens and hidden_dim=1024, the arithmetic intensity per weight byte is 64 multiply-accumulate operations — 64 ops/byte. This is high enough to approach the compute roof, which is why prefill is close to compute-bound. Fix class A for memory-bound decode: cache blocking, software prefetching, and ultimately increasing the effective batch size (process multiple tokens per step instead of one) to amortize weight loading cost. More arithmetic per byte loaded. Fix class B for compute-bound prefill: SIMD widening if moving to AVX-512 capable hardware, kernel fusion to reduce intermediate memory writes, and reducing per-layer overhead (sampling, normalization accounting for the remaining 16% of prefill time). The optimization roadmap is clear: the most impactful near-term work is on decode, in the vec_dot kernel, targeting cache blocking.

C safety in practice: deterministic inputs, instrumentation, and regression gates.

Performance optimization in C has a fundamental tension: the same techniques that make code faster — manual memory management, SIMD intrinsics, custom allocators, pointer arithmetic — are also the techniques most likely to introduce memory errors, undefined behavior, and non-deterministic output. A 5% speedup that ships with an out-of-bounds write or a use-after-free is not an optimization. It is a deferred crash. So the safety tooling layer is not optional — it is what makes performance claims trustworthy. The first command — profile-v7-prepare-runtime with V7_FORCE_COMPILE=1 — builds the inference runtime with debug symbols (-g flag) and deterministic inputs. We use this build for all profiling so that flamegraph function names are complete and call stacks resolve correctly. A profiling session against a stripped binary is much harder to read. The second command — profile-v7-cachegrind — runs Valgrind's cache simulator. Valgrind does not run your code natively — it interprets it through a virtual CPU model. This makes it approximately 20 to 50 times slower than native execution. But it has two important capabilities that sampling profilers lack: it counts cache events exactly rather than probabilistically, giving you precise miss counts per function, and it can detect memory errors including reads before writes, invalid address accesses, and heap corruption. For reduced token workloads like --max-tokens 16, Valgrind becomes practical even at 50x overhead because a short inference is still manageable in absolute time. The sanitizer pass — AddressSanitizer in particular — is the safety gate before performance testing. ASan instruments every memory access at compile time and reports any out-of-bounds reads or writes immediately with a full stack trace. For optimization patches that modify AVX SIMD hot paths, always run ASan on a small prompt before submitting the change for performance benchmarking. The principle: C stays safe when every optimization change goes through deterministic inputs, instrumented validation, and regression gates. Safety is not a separate workflow — it is woven into the profiling loop.
