Chapter 1

Derivatives are the language of tiny changes

What changes?

The model output changes when weights, inputs, or intermediate values change.

Why it matters

Training is repeated adjustment. Without derivatives, we do not know which direction to adjust.

slope = change in output / change in input
The first thing we need is the idea of a derivative. A derivative answers a very practical question: if I change this input by a tiny amount, how much does the output change? In machine learning, the input we change is often not the data itself. It is usually a weight, a bias, or an intermediate value inside the model. So the derivative tells us how sensitive the loss is to that specific value. If the derivative is positive, increasing the value may increase the loss. If the derivative is negative, increasing the value may decrease the loss. This is the basic signal that lets training move in a useful direction.
Chain rule

Backprop is bookkeeping for dependent changes

x
->
u = wx+b
->
y_hat
->
loss

If loss depends on y_hat, and y_hat depends on w, the chain rule tells us how w affected the loss.

The next idea is the chain rule. The chain rule is important because models are not one isolated operation. The input flows into wx plus b, the result becomes a prediction, and the prediction is compared against the target to produce a loss. So when the loss changes, we need to trace that effect backward through the dependency chain. If the loss depends on the prediction, and the prediction depends on the weight, then the loss also depends on the weight. Backpropagation is basically organized bookkeeping for this dependency graph. It tells us how every parameter contributed to the final error.
Chapter 2

The smallest training loop: wx+b

y_hat = w*x + b

One weight, one bias, one prediction, one loss. This is the cleanest place to understand gradient descent.

Existing blog asset: data, line, and prediction error.
Now we can place those ideas inside the smallest useful model: y hat equals w times x plus b. Here w is the weight, b is the bias, x is the input, and y hat is the prediction. This equation is simple enough that we can understand every piece of it, but it is not a toy in the wrong sense. It is the scalar version of what neural networks do at much larger scale. The model makes a prediction, the prediction has an error, and the training loop adjusts w and b to reduce that error. This is why wx plus b is such a good starting point. It lets us understand training without hiding the process behind a huge model.
Training loop

Forward, loss, backward, update

->
measure error
->

->
update w,b
->
repeat
y_hat = w * x + b
loss = mean((y_hat - y) ** 2)

dw = dloss_dw(...)
db = dloss_db(...)

w -= lr * dw
b -= lr * db
The training loop has four main parts. First, the model predicts. Second, we measure the error with a loss function. Third, we compute gradients so we know how each parameter affected the loss. Fourth, we update the parameters in the direction that should reduce the loss. Then we repeat the loop. This code is deliberately small, but the structure does not disappear when we move to transformers. The tensors get bigger, the kernels get faster, and the model has many more parameters, but the basic loop is still forward pass, loss, backward pass, and update.
Chapter 3

Matrix Wx+b is the same idea, scaled to layers

A neural network layer is many wx+b operations organized as matrix multiplication.

X [T x D] * W [D x H] + b [H] = Y [T x H]
Once wx plus b makes sense for one input, the next step is to scale it into matrices. Instead of one input value, we now have many tokens or many rows. Instead of one weight, we now have a weight matrix. The equation is still the same idea: input times weight plus bias. But now the shape contract matters. If X has shape T by D, and W has shape D by H, then the inner D dimensions match, and the output has shape T by H. This is why tensor shapes are not just notation. Shapes tell us whether the computation is even legal.

This is where the transformer starts to become less intimidating. A transformer block contains several linear projections. The Q, K, and V matrices are created with linear maps. The output projection after attention is another linear map. The MLP block is also built from matrix multiplies, often expanding the feature dimension and then compressing it back down. So when people talk about a transformer, a lot of the computation is still repeated matrix multiplication. From the C-Kernel-Engine perspective, this means GEMM is not just a math symbol. GEMM is a kernel contract with memory layout, tiling, cache behavior, and vectorization.
Chapter 4

Softmax turns scores into probabilities

p_i = exp(z_i) / sum_j exp(z_j)

Softmax keeps the relative ordering of scores but forces the outputs to sum to 1.

After linear layers produce scores, we often need to turn those scores into probabilities. That is the job of softmax. Softmax takes a list of logits, exponentiates them, and divides each value by the total sum. The result is a probability distribution. The values are all positive, and they sum to one. This operator appears in the final layer of classifiers, and it also appears inside attention. For this part of the talk, I want to focus on softmax as an operator. Attention will come later, but attention depends heavily on understanding this probability step.
Softmax backward

The full Jacobian exists, but kernels use the efficient form

Math view

J_ij = p_i * (delta_ij - p_j)

Each output probability is coupled to every input logit.

Kernel view

dx = p * (g - dot(p, g))

Same gradient, but computed in O(N) without materializing the matrix.

Softmax is also a good example of the difference between the mathematical derivation and the efficient implementation. Mathematically, the derivative of softmax is a full Jacobian matrix. Each output probability depends on every input logit, so the gradient is coupled. But in code, we usually do not want to materialize that full matrix. We can compute the same result with a more efficient expression: p times g minus the dot product of p and g. This is the pattern I want to keep using throughout the series. First derive the real math. Then implement the version that is efficient enough to run.
Chapter 5

Activations make stacked linear layers useful

Without nonlinear activations, stacking many linear layers collapses into one linear map.

ReLU, sigmoid, tanh, GELU, Swish, and SwiGLU each change the signal in a different way.

If we only stack linear layers, the whole stack can collapse into one larger linear map. That means depth alone is not enough. We need nonlinear activation functions to let the model represent more complex relationships. Different activations shape the signal differently. Sigmoid and tanh squash values into a bounded range. ReLU passes positive values and clips negative values. GELU and Swish are smoother functions that show up in many modern neural networks. SwiGLU adds a gating structure that is common in transformer MLP blocks. So activations are not decorations. They are one of the reasons the model can represent nonlinear behavior.
Activation gradients

Forward shape controls backward flow

Sigmoid/tanh

Saturate. Gradients can get small.

ReLU

Cheap and sparse, but can die below zero.

GELU/SwiGLU

Smoother gating behavior used in many transformer-style MLPs.

The activation function also controls how gradients flow backward. Sigmoid and tanh can saturate, which means the derivative becomes very small when the input is far from the center. ReLU is simple and fast, but if a unit stays below zero, its gradient can become zero. GELU and SwiGLU are smoother, and they often work better in transformer-style MLPs because they gate information in a less abrupt way. So when we choose an activation, we are choosing both a forward transformation and a backward gradient behavior. That is why this topic belongs in the training story, not just the inference story.
Chapter 6

LayerNorm and RMSNorm stabilize the signal

As layers stack, signal scale can drift. Normalization gives each block a more predictable input distribution.

LayerNorm: normalize by mean and variance
RMSNorm: normalize by root mean square
As models get deeper, the scale of the signal can drift from layer to layer. One block might produce values that are too large. Another block might produce values that are too small. If this keeps happening across many layers, training becomes harder. LayerNorm and RMSNorm help stabilize the signal. LayerNorm subtracts the mean and divides by the standard deviation. RMSNorm skips the mean subtraction and normalizes by the root mean square. The exact formulas differ, but the purpose is similar: give the next block a more predictable input scale. This is one of the reasons modern transformer stacks can be very deep.
Pre-LN vs Post-LN

Where the norm sits changes training behavior

Practical takeaway

Modern transformer stacks often prefer normalization before the sublayer because it gives gradients a cleaner highway through the residual path.

Normalization is not only about the formula. Placement also matters. In a Post-LN block, the normalization happens after the sublayer and residual connection. In a Pre-LN block, the normalization happens before the attention or MLP sublayer. Modern transformer stacks often prefer Pre-LN style designs because the residual path gives gradients a cleaner route backward through the network. I like to think of this as a gradient highway. If the residual path stays clean, the model can train more reliably as we add more layers. This is the point where we move from isolated math operators into real architecture decisions.
Chapter 7

Skip connections are gradient highways

Without skip connections, gradients must pass through every sublayer. Deep networks starve.

output = sublayer(x) + x

The residual path gives gradients a clean highway from loss to input, regardless of depth.

Why the clean path matters

In backprop, the + operator copies the gradient to both branches. One branch always carries dL/dx directly — no matrix multiply, no activation derivative, no norm scaling. This is why 100+ layer networks can train.

Skip connections, also called residual connections, are one of the most important architectural innovations in deep learning. Without them, gradients must pass through every sublayer during backpropagation. Each sublayer multiplies the gradient by its local Jacobian, and over many layers, those multiplications can shrink the gradient exponentially. This is the vanishing gradient problem. The skip connection fixes this by adding the input directly to the sublayer output. During the backward pass, the addition operator copies the gradient to both branches. One branch goes through the sublayer backward. The other branch passes the gradient directly through — no matrix multiply, no activation derivative, nothing. This creates a clean gradient highway that lets the loss signal reach early layers without degradation. This is why modern transformers can stack 80, 100, or even 200 layers and still train stably.
Pre-Norm residual

The clean path cannot be polluted

❌ Post-LN encapsulates the skip

x + Norm(sublayer(x))

The norm wraps the residual. Gradient must flow through the norm to reach the highway.

✓ Pre-LN keeps the highway clean

x + sublayer(Norm(x))

The norm is inside the sublayer branch. The residual path is a direct identity addition.

This is the difference between the 1986 backprop paper and deep transformer stacks. The clean residual path is a critical recipe for training stability.

This slide makes the key architectural point about skip connections. The clean residual path cannot be encapsulated or polluted. In a post-LN design, the normalization wraps the residual addition. That means the gradient highway passes through the norm's backward computation, which can attenuate or distort the signal. In a pre-LN design, the normalization sits inside the sublayer branch. The residual path is a direct identity addition — the gradient flows through unchanged. This seems like a small architectural choice, but it made the difference between training stability and training collapse in deep networks. It is one of the critical recipes that enabled the jump from shallow networks to very deep transformer stacks. This is also the difference between the 1986 backpropagation paper's practical depth limitations and today's 100-plus-layer models.
Chapter 8

Tokenization: text becomes numbers

A language model never sees text. It sees integer token IDs. Tokenization is the translation layer.

"Hello world"
->
tokenizer
->
[15496, 995]
Before anything else happens in a language model, text must become numbers. The model never sees characters or words directly. It sees integer token IDs that index into a vocabulary. Tokenization is the translation layer that converts raw text into these IDs. The quality of this translation matters more than most people realize. If the tokenizer splits a common word into too many pieces, the model wastes capacity reassembling it. If the vocabulary is too small, rare words get fragmented. If the vocabulary is too large, the embedding table becomes enormous. So tokenization is not just preprocessing — it is a design decision that affects model capacity, training efficiency, and inference cost.
Tokenization methods

From WordPiece to BPE to SentencePiece

BPE (GPT-2, GPT-4)

Start with characters, greedily merge the most frequent pair. Repeat until vocabulary size is reached.

WordPiece (BERT)

Similar to BPE but selects merges by likelihood gain, not raw frequency.

SentencePiece (LLaMA, Qwen)

Language-agnostic, operates on raw bytes. No pre-tokenization rules needed.

The three main tokenization methods each solve the same problem differently. BPE, which is what GPT-2 and GPT-4 use, starts with individual characters and greedily merges the most frequent adjacent pair. After enough merges, common words become single tokens and rare words stay as subword pieces. WordPiece, used in BERT, is similar but selects merges based on likelihood gain rather than raw frequency. SentencePiece, used in LLaMA and Qwen, operates directly on raw bytes and does not need language-specific pre-tokenization rules. I implemented a WordPiece tokenizer in C for BERT models during my work at BC Gov, and that experience later informed the tokenization approach in C-Kernel-Engine. The key lesson is that tokenization is deterministic — no trainable parameters — but the vocabulary design deeply affects everything downstream.
When to skip standard tokenization

Sometimes custom tokens are the right answer

For SVG training in C-Kernel-Engine, BPE was unnecessary. Custom tokens mapped SVG commands directly to vocabulary IDs.

The decision depends on whether natural language statistics match your domain. For structured formats, custom tokenization can outperform general-purpose BPE.

📺 Deep dive on @shivasnotes 📝 SVG Training Report
This is an important practical lesson from C-Kernel-Engine. When training on SVG data, we found that BPE and SentencePiece were not the right tools. SVG is a structured format with known commands like M, L, C, Z, and numeric coordinates. Using a standard text tokenizer would fragment these commands into arbitrary subword pieces that have no meaning in the SVG domain. Instead, we created custom tokens that mapped SVG path commands and coordinate ranges directly to vocabulary IDs. This gave the model a vocabulary that matched the structure of the data, which improved both training convergence and generation quality. The broader lesson is: tokenization is a design decision, not just a preprocessing step. If your data is natural language, BPE or SentencePiece is usually the right choice. If your data is structured, custom tokenization can be significantly better.
Chapter 9

Embeddings: token IDs become dense vectors

embedding[token_id] → vector ∈ ℝ^d

A learned lookup table. Each token ID indexes a row in the embedding matrix. No multiplication — just a table read.

Token 15496 → [0.12, -0.34, 0.87, ...] (d_model values)
Token 995 → [-0.45, 0.22, 0.01, ...] (d_model values)

Why not one-hot?

A one-hot vector for a 50K vocabulary would be 50,000 dimensions — almost entirely zeros. Dense embeddings compress meaning into d_model ≈ 768–4096 dimensions.

Backward pass

The gradient for embeddings is a scatter-add: accumulate dL/d(embedding) into the rows that were used by the input tokens.

After tokenization converts text into integer IDs, the model needs continuous vectors to compute with. The embedding layer is a learned lookup table. Each token ID indexes a row in the embedding matrix, and that row is the token's dense vector representation. There is no matrix multiplication here — just a table read. The reason we use dense embeddings instead of one-hot vectors is dimensionality. A one-hot vector for a 50,000-token vocabulary would have 50,000 dimensions, almost entirely zeros. Dense embeddings compress meaning into a much smaller space, typically 768 to 4096 dimensions. During training, the backward pass for embeddings is a scatter-add operation. We accumulate the gradient from the loss into the specific embedding rows that were used by the input tokens. Only those rows get updated — the rest of the table stays unchanged for that training step.
Chapter 10

The MLP block: per-token nonlinear processing

x
->
W₁x + b₁
->
activation
->
W₂(·) + b₂
->
output

The MLP expands the feature dimension (often 4×), applies a nonlinearity, then compresses back down. Each token is processed independently.

Shape flow

x [T × d] → W₁ [d × 4d] → act → W₂ [4d × d] → out [T × d]

Two GEMM operations plus activation. This is where most of the model's parameters live.

The MLP block is the other major component of a transformer layer, alongside attention. It processes each token independently through a two-layer feedforward network. The first linear layer expands the feature dimension, typically by a factor of four. Then a nonlinear activation function is applied — in modern transformers this is usually SwiGLU or GELU. Then the second linear layer compresses the dimension back down to the model dimension. So the MLP is expand, activate, compress. The key insight is that the MLP operates per-token. Attention mixes information between tokens. The MLP transforms each token's representation independently. Together, they give the transformer its power: attention routes information, and the MLP processes it. The MLP is also where most of the model's parameters live, because the 4x expansion creates large weight matrices.
SwiGLU MLP

Modern transformers gate the MLP path

Classic MLP

h = activation(xW₁ + b₁)
out = hW₂ + b₂

Simple expand → activate → compress. Used in GPT-2.

SwiGLU MLP

gate = swish(xW_gate)
up = xW_up
h = gate ⊙ up
out = hW_down

Two parallel projections: one gates, one carries. Used in LLaMA, Qwen, Mistral.

The gating mechanism lets the model learn which features to pass and which to suppress — a learned information filter per token.

Modern transformers use a gated MLP variant called SwiGLU. In the classic MLP, there is one expansion matrix, one activation, and one compression matrix. SwiGLU changes this by using two parallel projections: a gate projection and an up projection. The gate projection goes through a swish activation. The up projection passes the values directly. Then the two are multiplied elementwise. This means the gate controls which features in the up projection are allowed through. It is a learned information filter. The result goes through a down projection to compress back to the model dimension. This is three matrix multiplies instead of two, but the gating behavior gives the model more control over information flow. In C-Kernel-Engine, SwiGLU has a paired backward kernel because the elementwise gate multiplication requires both the gate output and the up output to compute gradients.

Chapter 11

Positional encoding tells attention where tokens are

Self-attention can compare tokens, but token order must be injected into the representation.

token meaning = embedding + position signal
modern path: RoPE rotates Q and K by position
Before we derive attention, we need one more idea: position. A transformer sees a sequence of token vectors, but plain self-attention does not automatically know whether a token came first, last, or somewhere in the middle. If we only give the model token embeddings, then the content is present, but the order is under-specified. Positional encoding fixes this. In older transformers, this could be sinusoidal position vectors or learned position embeddings. In many modern transformer-style models, the important idea is RoPE, or rotary positional embedding. RoPE does not simply add a position vector to the token. It rotates the query and key vectors according to position. That means the dot product between Q and K can carry relative position information. So when we move into attention, remember that Q and K are not just content vectors. They can also carry position through this rotation step.
Chapter 12

Attention is token-to-token routing

Each token asks: which previous tokens should influence my next representation?

Q
·
K^T
->
scores
->
weights
->
mix V
Now we can explain attention directly. Attention is token-to-token routing. For each token, the model asks: which other tokens should influence this token's next representation? The query vector is the question that the current token asks. The key vector is what each token offers for matching. The value vector is the information that will actually be copied or mixed if the attention weight is high. The score matrix comes from Q times K transpose. Then we scale the scores, apply the causal mask if this is a decoder, and run softmax. The softmax output gives us attention weights. Finally, those weights multiply V, which produces the mixed representation. So attention is not one magical operation. It is a pipeline: project Q K V, compare Q and K, normalize scores into weights, and use those weights to mix V.
Multi-head attention

Many heads, many perspectives

Why multiple heads?

Each head operates in a smaller subspace (d_model / n_heads). Different heads can learn different attention patterns — syntax in one, semantics in another, positional in a third.

Grouped Query Attention (GQA)

Share K/V across groups of Q heads. Reduces KV-cache memory during inference without significant quality loss. Used in LLaMA 2+, Qwen, Mistral.

Multi-head attention splits the model dimension into multiple heads, each operating in a smaller subspace. If the model dimension is 4096 and we have 32 heads, each head works with 128 dimensions. This is important because different heads can learn different kinds of attention patterns. Some heads may focus on syntactic relationships, others on semantic similarity, and others on positional proximity. Grouped Query Attention is an optimization where multiple query heads share the same key and value heads. This reduces the KV-cache memory during inference — which matters a lot for serving long-context models — without significant quality loss. GQA is used in LLaMA 2, Qwen, and Mistral.
Attention forward pass

The complete forward path is a sequence of known operators

This slide is the complete attention forward pass at the operator level. We start with X, the token representations entering the block. We create Q, K, and V using three learned linear projections. Then we split those projections into heads, because each head gets its own lower-dimensional attention space. If the model uses RoPE, we rotate Q and K by position. Then we compute Q times K transpose, divided by the square root of the head dimension. The division by square root d keeps the scores from becoming too large as the head dimension grows. In a decoder, we apply the causal mask so each token cannot look into the future. Then softmax converts the masked scores into attention weights. Those weights multiply V, producing the per-head output. Finally, we concatenate the heads and apply the output projection W O. Every part of this forward pass is something we can map back to a blog topic and a kernel: GEMM, RoPE, masking, softmax, value mixing, and projection.
Why attention gets expensive

The score matrix grows with context length squared

Shape pressure

scores = Q [T x d] * K^T [d x T] = [T x T]

Long context makes the attention matrix huge.

Kernel pressure

Materializing scores and softmax weights can dominate memory, especially during training when backward needs saved values.

The main reason attention becomes expensive is the score matrix. If the sequence length is T, then Q times K transpose produces a T by T matrix for each head. That means the memory and compute pressure grows with context length squared. For short sequences, this is manageable. For very long sequences, it becomes the bottleneck. This is why attention kernels matter so much. It is also why modern models keep inventing attention variants: grouped query attention, sliding window attention, compressed attention, sparse attention, and flash attention. The goal is usually the same: preserve the useful behavior of attention while avoiding the cost of fully materializing enormous T by T matrices.
Flash Attention intuition

Do the same math without storing the whole matrix

Flash Attention is not a different model idea. It is a better implementation strategy for exact attention.

Tile Q, K, and V. Stream blocks through cache. Keep an online softmax state. Avoid writing the full attention matrix to memory.

Flash Attention is important because it separates the mathematical model from the implementation strategy. The model is still doing attention. The output is still based on Q, K, V, softmax, and value mixing. The difference is that the kernel avoids writing the full attention matrix to memory. Instead, it processes Q, K, and V in tiles. It keeps track of the softmax normalization online, so it can stream through blocks and still get the correct result. This is exactly the kind of idea I want to understand in C-Kernel-Engine terms. The math says attention. The kernel asks: what must be stored, what can be recomputed, what can stay in cache, and what must be saved for backward?
Chapter 13

The loss starts the backward pass with p minus one-hot

dlogits = p - one_hot(target)

Softmax plus cross entropy collapses into a clean gradient at the output row.

Now we move from forward inference into training. At the output of the model, we have logits over the vocabulary. Softmax turns those logits into probabilities. Cross entropy compares those probabilities against the target token. The beautiful result is that the gradient with respect to the logits becomes p minus one hot of the target. This is one of the most important shortcuts in neural network training. We do not need to separately build the full softmax Jacobian and then multiply by the cross entropy derivative. The combined operator gives us a clean row of gradients. The correct target logit receives probability minus one. Every other logit receives its predicted probability. That row is the starting signal for the full backward pass through the language model.
Full backward pass

Backprop reverses the whole model graph

loss
->
LM head
->
final norm
->
decoder layers reversed
->
embedding scatter-add
The full backward pass is the chain rule applied to the entire model graph in reverse. We start from the loss gradient at the logits. Then we go backward through the LM head, which is usually a matrix multiplication against the vocabulary weights. Then we pass backward through the final normalization. After that, we walk through the decoder layers in reverse order. Inside each decoder layer, we reverse the residual additions, the MLP, the attention block, the Q K V projections, RoPE, softmax, and value mixing. Finally, when the gradient reaches the token embeddings, we scatter-add gradients into the embedding table rows that were used by the input tokens. This is why backprop is not one formula. It is a stitched sequence of local gradient rules. Each operator has a backward rule, and the training engine has to connect those rules in the correct order.
Decoder layer backward

Residual paths split and recombine gradients

The key idea

The residual path sends gradient directly backward, while attention and MLP paths compute their own parameter and input gradients.

A decoder layer backward pass is where the bookkeeping becomes more serious. In the forward pass, residual connections add sublayer outputs back into the main stream. In the backward pass, that addition becomes a gradient split. Part of the gradient flows directly through the residual path. Part flows into the sublayer that created the added output. For the MLP, that means reversing the second projection, the activation or gate, and the first projection. For attention, that means reversing the output projection, head concatenation, value mixing, softmax, score computation, RoPE, and Q K V projections. The important point is that every branch must contribute back to dX. The final input gradient for the layer is the sum of the residual gradient and the gradients from the attention and MLP paths.
Saved tensors

Backward needs evidence from forward

Attention

May need Q, K, V, weights, masks, or enough state to recompute them.

Norms

Need normalized input, scale, RMS or variance state.

Activations

Need input or output values to compute local derivatives.

One of the practical lessons from implementing training is that backward needs evidence from forward. The backward rule for an operator often depends on values that were computed during the forward pass. Attention may need Q, K, V, masks, softmax weights, or enough state to recompute them safely. Normalization may need the input, the normalized value, the scale parameter, and the RMS or variance state. Activation backward often needs either the original input or the activation output. This is why training memory is harder than inference memory. In inference, we mostly care about producing the next output. In training, we also need to preserve or recompute the information required to differentiate every operator. C-Kernel-Engine's save-for-backward rules make this explicit instead of hiding it inside a runtime autograd graph.
Static IR vs autograd

The difference is stitching, not magic

PyTorch style

Build a dynamic graph at runtime, then walk that graph backward.

C-Kernel-Engine style

Declare forward ops and grad rules, derive saved tensors, and generate the training path.

This is a point I want to say carefully. PyTorch is not magic, and C-Kernel-Engine is not trying to avoid the math. Both systems need forward kernels and backward kernels. Both systems need correct local gradient rules. The difference is how the pieces are stitched together. PyTorch usually builds a graph dynamically while the program runs, and autograd walks that graph backward. C-Kernel-Engine is more static. The IR declares the forward operations, the grad rules declare the backward contract, and the builder can derive which tensors must be saved for backward. So the difference is not that one system needs gradients and the other does not. The difference is whether the stitching is mostly runtime graph walking or compile-time generation from explicit contracts.
Chapter 14

Optimizers decide how parameters move

Gradients tell us direction. The optimizer decides the actual step.

new weights
Once backprop has computed gradients, the model still has not learned yet. Learning happens when the optimizer updates the parameters. The gradient tells us a direction, but the optimizer decides the actual step. Plain SGD takes a direct step opposite the gradient. Momentum adds velocity, so the optimizer can smooth noisy gradients and keep moving through shallow regions. Adam keeps moving averages of both the gradient and the squared gradient, which creates an adaptive step size per parameter. AdamW changes how weight decay is applied, decoupling it from the gradient update. So the optimizer is the bridge between gradient information and actual parameter movement.
SGD to AdamW

Each optimizer fixes a specific failure mode

OptimizerWhat it addsWhy it matters
SGDStep opposite gradientSimple, but noisy and sensitive to learning rate.
MomentumVelocity stateSmooths noise and helps push through flat regions.
AdamFirst and second momentsAdapts step size per parameter.
AdamWDecoupled weight decaySeparates regularization from gradient scaling.
I want to present optimizers as a sequence of fixes, not as a list of names. SGD is the simplest rule: move weights opposite the gradient. The problem is that raw gradients can be noisy, and the same learning rate may not work well for every direction. Momentum adds a velocity state, which smooths the path and helps the update keep moving through flat or noisy regions. Adam adds two moving averages: the first moment tracks the gradient direction, and the second moment tracks the scale of recent squared gradients. That lets Adam adapt the step size for each parameter. AdamW fixes a subtle but important issue: weight decay should not be entangled with Adam's adaptive gradient scaling. By decoupling weight decay, AdamW gives a cleaner regularization behavior and is widely used in transformer training.
AdamW kernel view

The update is also a performance-critical kernel

m = beta1 * m + (1 - beta1) * grad;
v = beta2 * v + (1 - beta2) * grad * grad;
update = m_hat / (sqrt(v_hat) + eps);
param = param - lr * update;
param = param - lr * weight_decay * param;
AdamW also matters as a kernel. For every trainable parameter, the optimizer reads the parameter, the gradient, the first moment, and the second moment. It updates the moving averages, applies bias correction or an equivalent corrected learning rate, computes the normalized update, applies the parameter step, and applies decoupled weight decay. That is a lot of memory traffic. For large models, optimizer state can be much larger than the raw parameter tensor. This is why fused optimizer kernels matter. If we can combine gradient clipping, moment updates, parameter updates, and weight decay into one pass, we reduce memory traffic and improve throughput. Again, the math is one part of the story. The memory movement is the other part.
The complete loop

Now the whole training system is visible

One loop

Tokenize, embed, run decoder layers, compute logits, compute loss, backpropagate, update weights, repeat.

At this point, the full training system is visible. We start with text and tokenize it. We look up embeddings and add position information. We run the tokens through decoder layers made of normalization, attention, residual paths, and MLPs. We project to vocabulary logits. We compute softmax cross entropy loss. The backward pass sends gradients from the loss through the LM head, final norm, decoder layers, attention, MLP, and embeddings. Then the optimizer updates the parameters. And the whole loop repeats. This is the bridge from basic machine learning to actual language model training. The pieces are still the same: linear maps, softmax, activations, norms, attention, gradients, and optimizer updates. The difference is scale, memory layout, kernel implementation, and the discipline required to make every step correct.
