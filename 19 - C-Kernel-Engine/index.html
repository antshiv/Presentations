C-Kernel-Engine

Kernel-First Transformer Engine in C

From 15k-Line Monoliths to Small, Verified Kernels

Open with the story: C-Transformer was a 15k-line C codebase with a hidden training bug; C-Kernel-Engine is the clean reboot that keeps what worked and fixes what didn’t. Emphasize the shift from “one giant training file” to “many small, validated kernels plus a DSL,” and set the expectation that this talk is about engineering lessons, not just model hype. Transition: tell them this first section explains why a new engine was necessary at all.

Why C-Kernel-Engine?

Problems in C-Transformer

  • 15k-line monolithic main.c
  • Training bug hidden in huge codebase
  • Forward / backward split across files and flags
  • Debug builds diverged from production paths

What Worked Well

  • Inference validated against PyTorch (< 1e-5)
  • AVX-512 + OpenMP speedups
  • Strong validation tooling in Python
  • Clear patterns across layers and models
Goal: Keep everything that worked (inference, validation, performance) and redesign the architecture so training and debugging stay tractable.
Frame this slide as “what I refuse to give up” (PyTorch-validated inference, performance work, Python tooling) versus “what I never want again” (monolith, hidden training bug). Key takeaway: this is not a rage-rewrite; it’s a deliberate redesign to make training and debugging boring and predictable. Transition: next, show the explicit design goals that fall out of these good/bad lists.

Design Goals

Architecture

  • Small, composable kernels (200–500 lines)
  • Graph-based DSL for model composition
  • Clear separation: Kernels / DSL / Models
  • Shared kernel set across GPT, Qwen, LLaMA

Engineering Practice

  • Forward + backward implemented together
  • Immediate validation vs PyTorch per kernel
  • Built-in debug modes per kernel, not global
  • End-to-end training checks at every milestone
Call out the two columns explicitly: architecture choices (kernels, DSL, shared across models) and engineering process (forward+backward together, immediate validation, proper end-to-end tests). In plain English: “We’re designing so bugs are local and obvious instead of global and mysterious.” Transition: now that goals are clear, show how they map to a three-layer architecture.

Architecture Overview

graph TB subgraph "Kernel Layer" K1[RMSNorm] K2[RoPE] K3[MatMul] K4[Softmax] K5[GQA] K6[SwiGLU] end subgraph "DSL Layer" D1[Graph Builder] D2[Graph Executor] D3[Graph Optimizer] end subgraph "Model Layer" M1[GPT-2] M2[Qwen-2.5] M3[LLaMA-3] end K1 --> D1 K2 --> D1 K3 --> D1 K4 --> D1 K5 --> D1 K6 --> D1 D1 --> D2 D2 --> D3 D3 --> M1 D3 --> M2 D3 --> M3

Kernels know math and memory; DSL knows graphs and execution; models are just declarations.

Walk top to bottom: kernels implement low-level math, the DSL builds and runs graphs, and models become thin configuration on top. Emphasize that most work and complexity now lives in the bottom two layers, so adding a new model is “just wiring up a graph.” Transition: from this high-level picture, zoom into what a single kernel is responsible for.

Kernel Design

Per-Kernel Responsibilities

  • Forward and backward functions
  • Shape and type checks
  • Memory access pattern documented
  • Numerical stability (epsilon, scaling)

Per-Kernel Tooling

  • Unit tests vs PyTorch reference
  • Debug mode with extra asserts
  • Small benchmarks (throughput, latency)
  • Optional AVX/OpenMP specializations
Rule: Never promote a kernel to "stable" until it passes forward + backward validation and basic perf checks.
Contrast directly with the old approach: before, large subsystems were written before validation; now each kernel has a clear contract, tests, and benchmarks before it’s “real.” Key idea to say out loud: “If a bug appears, it must be in one kernel or in the graph, never in a 15k-line soup.” Transition: with that in mind, show a concrete example kernel: RMSNorm.

RMSNorm: Forward Pass

graph LR X[Input x] --> SQ[Square] SQ --> MEAN[Mean] MEAN --> RSQ[Inv sqrt] RSQ --> NORM[Normalize] NORM --> SCALE[Scale gamma] SCALE --> Y[Output y]

One pass over activations, simple horizontal dataflow, perfect for FMA and vectorization.

Explain RMSNorm in human terms: we look at the average energy of the vector (mean of squares), normalize, then scale with gamma, all in a single left-to-right sweep. Emphasize that this straight-line structure is exactly what C-Kernel-Engine likes: one kernel, one pass, ideal for SIMD/FMA and easy to debug. Transition: on the next slide, show how the backward pass is just this graph “unwound” into chained gradients.

RMSNorm: Backward Pass

graph LR DY[grad_y] --> DG[grad_gamma] DY --> DN[grad_norm] DN --> DRSQ[grad_rsqrt] DRSQ --> DMEAN[grad_mean_x2] DMEAN --> DX2[grad_x2] DX2 --> DX[grad_x]

Backward is expressed as a chain of simple ops we can validate numerically against PyTorch.

Make it clear that nobody in the audience needs to memorize these gradient names; what matters is the shape: a short chain where each step has a small, testable formula. Connect this back to your validation philosophy: you can numerically check each step against PyTorch or finite differences, so there’s never a mystery gradient. Transition: now that they’ve seen one kernel in detail, move up to a more complex structure: multi-head attention.

Multi-Head Attention: Forward

graph LR X[Input] --> PROJ[Linear to Q K V] PROJ --> Q[Q] PROJ --> K[K] PROJ --> V[V] Q --> SCORES[Scores] K --> SCORES SCORES --> MASK[Mask] MASK --> P[Softmax] P --> WEIGHTED[Weighted V] V --> WEIGHTED WEIGHTED --> OUT[Output proj]
Describe attention structurally: project once to get Q/K/V, compute scores, apply masks, softmax, then mix values back together and pass through an output projection. Stress that the DSL sees exactly this graph, which means attention becomes “just another composition of kernels,” not a special hard-coded block. Transition: highlight that even nonlinear bits like GELU and SwiGLU are treated the same way—simple graphs over kernels.

Activation Kernels

GELU Forward

graph LR X[Input] --> SCALE[Scale] SCALE --> ERF[Erf] ERF --> COMB[Combine] COMB --> Y[Output]

SwiGLU Forward

graph LR X[Input] --> SPLIT[Split] SPLIT --> A[Gate part] SPLIT --> B[Value part] A --> SW[Swish] SW --> PROD[Gate * value] B --> PROD PROD --> Y[Output]
Use this slide to demystify the activations: on paper GELU and SwiGLU look fancy, but in the engine they’re just a few nodes—scale, erf, combine, split, swish, multiply. Call out that this makes it obvious where to fuse work (e.g., combine activation with surrounding matmuls) and that the backward pass is just local derivatives plus the chain rule through this tiny graph. Transition: from individual kernels and tiny graphs, move back up to how the DSL manages full-model graphs.

DSL & Graph Execution

Graph Builder

  • Declarative API to connect kernels
  • Handles tensor lifetimes and aliases
  • Records shapes and strides
  • Exports graph for visualization (Mermaid, DOT)

Executor & Optimizer

  • Topological scheduling with minimal buffers
  • Kernel fusion opportunities (e.g., RMSNorm + MatMul)
  • Backend-aware (AVX, plain C, embedded)
  • Tracing hooks for debugging and profiling
Explain that the graph builder is how you write “what happens” in the model: add nodes, wire tensors, record shapes—nothing about threads or AVX here. Then the executor/optimizer decide “how it runs”: scheduling, memory reuse, fusion, and backend choice, with hooks so you can trace any subgraph. Transition: now you can compare life before and after this architecture to show why it will behave better than the original C-Transformer.

Fused Kernels & oneDNN Lessons

graph LR SPEC[Op spec] --> PLAN[Kernel plan] PLAN --> IMPL1[Plain C] PLAN --> IMPL2[Vectorized] PLAN --> IMPL3[Embedded] PLAN --> NOTES[Perf notes] NOTES --> REG[Kernel registry]

We borrow ideas (not code) from oneDNN: describe kernels once, specialize per backend, track what is fused.

Clarify that the “kernel plan” is where you describe tiling, layouts and FMA usage once, then generate or hand-write different implementations for plain C, SIMD, or embedded. Mention that the registry acts like a directory of kernels and fusions: you can see exactly which operations are optimized on which CPUs, instead of rediscovering the same ideas. Transition: with that infrastructure in place, show what concrete demos you want to build on top.

Every demo is built on the same kernels and DSL, so improvements propagate automatically.

Important line: “Every step reuses the same kernels and DSL, so once a kernel is fast and correct, all higher-level demos benefit automatically.” Transition: next, walk through how you’ll systematically make those kernels and graphs faster.

Performance Plan

graph LR BASE[Baseline] --> VEC[Vectorize] VEC --> FUSE[Fuse kernels] FUSE --> MEM[Better layouts] MEM --> DIST[Multi-core] DIST --> HW[Embedded]

The idea: always have a correct baseline, then layer optimizations step by step with benchmarks.

Emphasize the staircase: baseline correctness first, then vectorization, then fusions, then better layouts, then multi-core, then hardware-specific tweaks. Make the point that each step is gated by tests and microbenchmarks; you never “optimize blind” and reintroduce the kind of bug that killed C-Transformer’s training. Transition: with performance strategy clear, cover how you’re thinking about precision, quantization, and storage.

Numerical Precision & Storage

graph LR WQ[Quant weights] --> DEQ[Dequant to f32] DEQ --> MATMUL[Matmul f32] MATMUL --> ACT[Activations f32] ACT --> NORM[Norm] NORM --> OUT[Outputs]

Plan: start with f32 everywhere, then introduce quantized weights while keeping f32 activations and clear error budgets.

Say explicitly that step one is “boring but safe”: pure f32 everywhere, with clear validation against PyTorch, before any quantization. Then explain that you’ll introduce quantized weights gradually, keeping activations in f32 and tracking choices like int8 vs int4 and per-channel vs per-tensor scales in the graph metadata. Transition: finally, zoom back out to the high-level roadmap and risks so people see the whole journey.

Roadmap & Milestones

Phase 1: Core kernels (RMSNorm, RoPE, MatMul, Softmax)
Phase 2: Graph builder + executor prototype
Phase 3: GPT-2 inference via DSL (no training)
Phase 4: Step-by-step training loop vs PyTorch
Phase 5: Qwen-2.5 and LLaMA integration
Constraint: Never start the next phase until the current one passes validation (numerical + perf).
Highlight that progress is staged and verifiable. No more "big bang" integration at the end.

Risks & Open Questions

Technical Risks

  • Keeping DSL simple enough for C
  • Managing complex memory lifetimes safely
  • Balancing portability vs AVX-level tuning

Product Risks

  • Scope creep on supported models
  • Time spent on tooling vs core kernels
  • Maintaining good documentation as we move fast
Plan: Keep a narrow, high-value core (Qwen-2.5 inference) and treat everything else as optional extensions.
Be honest about uncertainty. Show that risk is recognized and managed, not ignored.
