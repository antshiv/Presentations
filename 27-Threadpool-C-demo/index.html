C-Kernel-Engine Runtime

Threadpools
From First Principles

A small C curriculum: raw pthreads → runtime structures → persistent workers → reusable dispatch → dummy GEMV workload.

Today I want to build threadpools from first principles. The hook is simple: a fast CPU kernel is not enough if the runtime keeps creating threads, allocating memory, or hiding scheduling decisions. I want to start with pthreads, then define the structures, then create the pool, then add dispatch, and finally connect it to a small matrix-vector workload. The point is to show how a C kernel becomes part of a reusable runtime.

The Presentation Flow

The hook

The presentation flow should feel natural. Start with the real problem: thread creation in the hot path is a runtime tax. Then show that the pthread API we need is smaller than it looks. Then build the threadpool in small files. Finally compile and run each stage so the audience sees the behavior and understands why it matters for AI kernels.

Why This Matters

Kernel

Computes math over pointers.

Threadpool

Owns work scheduling and barriers.

Runtime Contract

Keeps thread creation outside the hot path.

A CPU AI runtime needs the kernel and the dispatch contract.

The big picture is that a kernel alone is not a model runner. A kernel computes math. A threadpool decides which worker owns which rows or tokens, how workers sleep, and how the caller waits for completion. Today I am keeping the focus on the threadpool contract.

Why Not Just Use OpenMP?

OpenMP fork/join

Easy to write, but every parallel region can pay scheduling, wakeup, and barrier overhead.

Threadpool dispatch

Create workers once, reuse them thousands of times, and send small kernel jobs quickly.

LLM decode loop:
  token 0:
    RMSNorm dispatch
    QKV GEMM dispatch
    attention dispatch
    MLP GEMM dispatch
    optimizer/training dispatch ...

If each small op forks a new team, overhead becomes the workload.
This is not an attack on OpenMP. OpenMP is useful. The problem is using fork-join parallel regions inside a tight AI runtime loop where hundreds of small kernels run per token or per training step. If every little operation wakes or creates a team and then joins it again, the overhead can dominate. A persistent threadpool pays the worker creation cost once, then dispatches small jobs repeatedly.

Pthread API Map

Thread identity

pthread_t
opaque handle for one POSIX thread.

Create / join

pthread_create
pthread_join
start work, then wait for completion.

Coordination

pthread_mutex_t
pthread_cond_t
protect state and sleep until work arrives.

A threadpool is mostly these APIs assembled into a reusable runtime contract.

The pthread API surface we need is small. pthread_t identifies a worker. pthread_create starts it. pthread_join waits at shutdown. Mutexes protect shared pool state. Condition variables let workers sleep without busy waiting. A thread pool is mostly these pieces assembled into a reusable scheduling contract.

pthread_create Contract

int pthread_create(pthread_t *thread,
                   const pthread_attr_t *attr,
                   void *(*start_routine)(void *),
                   void *arg);
  • thread: where the OS stores the new worker handle.
  • attr: usually NULL for default attributes.
  • start_routine: worker entry function.
  • arg: one pointer passed into that worker.
This is the signature I care about from the man page. The start routine must be a function that takes void star and returns void star. The arg is one pointer. That is why real code usually creates a small worker argument struct containing the pool pointer and the worker index.

Condition Variable Pattern

pthread_mutex_lock(&p->mu);
while (my_generation == p->generation && !p->stop) {
    pthread_cond_wait(&p->has_work, &p->mu);
}
pthread_mutex_unlock(&p->mu);

The loop matters: condition variables can wake spuriously.

The mutex protects the predicate; the condition variable only sleeps and wakes.

This is the key synchronization pattern. A condition variable is not the state. The state is the predicate, in this case generation changed or stop requested. The mutex protects that predicate. pthread_cond_wait atomically unlocks the mutex and sleeps. When it wakes, it locks again. The while loop matters because wakeups can be spurious and because multiple workers wake at once.

2. Create The Structures

make run-stage2
typedef void (*work_fn_t)(int ith, int nth, void *arg);

typedef struct {
    pthread_t *threads;
    int nthreads;
    pthread_mutex_t mu;
    pthread_cond_t has_work;
    pthread_cond_t done;
    work_fn_t fn;
    void *arg;
    int generation;
    int completed;
    int stop;
} threadpool_t;
Now we create the runtime structure. The pool needs worker handles, synchronization objects, the function pointer, the opaque argument pointer, a generation counter, completion counter, and stop flag. This is the minimal state for dispatching work repeatedly. This file intentionally does not create threads yet; it makes the runtime state visible first.

3. Create Workers Once

make run-stage3
for (int i = 1; i < nthreads; ++i) {
    args[i - 1].pool = p;
    args[i - 1].ith = i;
    pthread_create(&threads[i - 1], NULL,
                   worker_main, &args[i - 1]);
}

Thread 0 is the main thread.

Workers are 1..nthreads-1.

A key design choice is that thread zero is the main thread. We create n minus one worker threads. The main thread should not just dispatch and sleep. It should also do its share of work during dispatch.

4. Dispatch Logic

make run-stage4
p->fn = fn;
p->arg = arg;
p->completed = 0;
p->generation++;
pthread_cond_broadcast(&p->has_work);

fn(0, p->nthreads, arg); // main thread works too

Workers wake when generation changes.

Dispatch writes the function and argument, resets completion, increments generation, and wakes workers. Then the main thread immediately runs the same work function as thread zero. This is the core of the reusable dispatch idea.

Run Stage 4

make run-stage4

thread 3/4 received dispatch

thread 1/4 received dispatch
...
second dispatch reuses the same pool
Stage four proves dispatch. Every thread receives work. Then the second dispatch happens without creating new threads. This is the central threadpool behavior.

5. Dummy GEMV Workload

make run-final
int r0 = (ith * rows) / nth;
int r1 = ((ith + 1) * rows) / nth;

for (int r = r0; r < r1; ++r) {
    float sum = 0.0f;
    for (int c = 0; c < cols; ++c)
        sum += w[r * cols + c] * x[c];
    out[r] = sum;
}
The final workload is a tiny GEMV shape. Each thread owns a row range. This maps directly to how many CPU kernels are partitioned: split output rows, make each thread write its own range, then wait at the barrier.

From Demo GEMV To AI Kernels

Demo

matvec_work() splits output rows across workers.

Real runtime

GEMV/GEMM/RMSNorm/attention kernels split rows, tokens, heads, or tiles.

Same contract

fn(ith, nth, arg) gives every worker its worker id and total workers.

Same barrier

The caller waits until all worker-owned ranges finish.

This is where I connect the toy demo to the AI runtime. The demo uses matrix-vector multiplication because it is easy to see. But the runtime idea is the same for GEMV, GEMM, RMSNorm, attention, softmax, and optimizer kernels. Each dispatch gives every worker an id, total worker count, and payload. The kernel decides what range or tile belongs to that worker. Then the runtime waits at a barrier.

How GEMM Dispatch Maps To Cores

// Runtime dispatch shape:
threadpool_dispatch(pool, gemm_work, &job);

// Worker range shape:
int row0 = (ith * M) / nth;
int row1 = ((ith + 1) * M) / nth;

// Each worker computes C[row0:row1, :]

For bigger kernels, the same idea becomes block/tile ownership instead of only row ownership.

For GEMM, the simplest mental model is row partitioning. Worker zero owns one row range, worker one owns another row range, and so on. For high-performance GEMM, this becomes tiled ownership because cache reuse matters. But the scheduling idea is the same: assign independent output regions to cores, keep writes separate, and synchronize after the dispatch.

LLM Runtime View

One model layer

RMSNorm → QKV GEMM → attention → output GEMM → MLP GEMM.

Many dispatches

Each operation becomes one or more threadpool dispatches.

Core ownership

Workers own rows, tokens, heads, blocks, or expert groups.

Runtime discipline

No hidden thread creation inside the kernel hot path.

In a real LLM runtime, one layer is not one operation. It is a chain of dispatches: normalization, QKV projection, attention, output projection, MLP projection, activation, and another projection. The threadpool is the execution substrate. Kernels should not secretly create new threads. The runtime owns the workers and dispatches each operation across cores.

Oversubscription

Good

One runnable worker per physical core, predictable affinity, enough work per dispatch.

Bad

Too many workers, nested OpenMP, allocator locks, cache-line fights.

More threads can make performance worse.

Oversubscription means creating more runnable threads than the hardware can use effectively. More threads do not automatically mean more speed. If each library secretly creates its own threads, the system can oversubscribe, fight over cores, and lose performance.

Threadpool Design Rules

  • Create workers once; dispatch many times.
  • Keep the hot path allocation-free.
  • Use a generation counter so workers do not repeat stale work.
  • Make thread 0 participate instead of only coordinating.
  • Use half-open ranges like [r0, r1) so partitions are exact.
These are the design rules I want the audience to keep. Create workers once. Dispatch many times. Keep the hot path allocation-free. Use a generation counter to identify fresh work. Make thread zero participate. Use half-open ranges so every row or token belongs to exactly one worker.
